# PRD — OpsBridge

> **OpsBridge** is an SSH-login agent daemon. SSHing into a dedicated user
> (`agent`) drops the operator directly into **pi.dev** — a terminal AI agent —
> instead of a shell. OpsBridge owns SSH integration, model initialization, and
> the admin CLI. Pi.dev owns everything else: the TUI, tools, and LLM calls.

## 1. Vision

A Linux host where `ssh agent@host` drops you directly into pi.dev's AI agent
TUI. The agent translates natural-language requests into shell commands on the
host using pi.dev's built-in tools (read, write, bash, grep, find, ls).

The product is the smallest possible "talk to my server in English" surface:
no web UI, no auth tokens to manage on the client side, no custom protocol.
Just `ssh agent@host`.

**Audience.** Self-hosting operators — solo DevOps and small teams running
their own bare-metal / VPS infrastructure. Not a smarter local terminal (that
space is well-covered by `aichat`, `aish`, `ai-shell`, etc.). Not for
managed / locked-down environments where SSH is not the operating surface.

## 2. User flow (single session)

1. Operator runs `ssh agent@host` from their laptop.
2. `sshd` authenticates them via public key (password auth disabled).
3. `sshd`'s `ForceCommand` runs `/usr/local/bin/opsbridge-agent`.
4. The launcher validates TTY, loads the API key, and `exec`s pi.dev with the
   configured model. The user's shell is never spawned.
5. Pi.dev starts its TUI. The operator types a request in English.
6. Pi.dev handles the full agent loop: LLM reasoning + tool execution.
7. Operator types another request, or exits via pi.dev's native quit gesture.

## 3. Architecture

```
        ┌──────────────────────────────────────────────────────┐
        │ Linux host                                           │
        │                                                      │
        │  sshd ── pubkey auth ──► ForceCommand                │
        │                               │                      │
        │                               ▼                      │
        │                  /usr/local/bin/opsbridge-agent       │
        │                  (bash script, generated by opsbridge)│
        │                  - TTY guard (exit 2 if non-TTY)     │
        │                  - export API key from api.key        │
        │                  - exec pi --model provider/model     │
        │                               │                      │
        │                               ▼                      │
        │                    ┌─────────────────┐               │
        │                    │    pi (pi.dev)  │──► LLM API    │
        │                    │  TUI + tools +  │               │
        │                    │  agent loop     │               │
        │                    └─────────────────┘               │
        │                                                      │
        └──────────────────────────────────────────────────────┘
```

Two components:

- **`/usr/local/bin/opsbridge-agent`** — a small bash script generated by
  `opsbridge install` / `opsbridge config`. It guards against non-TTY invocations,
  sets the API key env var from `/etc/opsbridge/agent/api.key`, and `exec`s pi
  with the configured model. No long-running daemon.
- **`opsbridge`** — admin CLI (Python, in its own venv). Not in the request
  path. Used by the host operator for install, health checks, and toggling
  access. See §7.

**Pi.dev.** The agent runtime is `@earendil-works/pi-coding-agent` (npm).
Pi.dev provides its own TUI, built-in tools, model routing, and session
management. OpsBridge delegates all of that to pi.

**System prompt.** OpsBridge writes a minimal safety-rules file to
`~agent/.pi/agent/SYSTEM.md` at install time. Pi.dev reads this path
automatically as its global system prompt — no CLI flag needed.

## 4. Tools

Pi.dev's built-in tools handle all agent operations:

| Tool | Purpose |
|------|---------|
| `read` | Read files |
| `write` | Write / overwrite files |
| `bash` | Execute shell commands |
| `edit` | In-place file edits |
| `grep` | Search file contents |
| `find` | Find files by name/pattern |
| `ls` | List directory contents |

No custom tools. No `remember` (cross-session preferences), no `ask`
(operator confirmation form), no `search`/`visit` (web retrieval) in the
launcher — these are either handled natively by pi.dev or outside scope.

## 5. Security model

- **Auth:** SSH public key only. `PasswordAuthentication no` for the `agent` user.
- **Privileges:** the `agent` user has `NOPASSWD: ALL` sudo, installed at
  `/etc/sudoers.d/opsbridge-agent` by `opsbridge install`. The model prefixes
  commands with `sudo` when root is needed.
- **Trust boundary:** possession of an SSH key authorized for `agent` is
  equivalent to root on this host. There is no second authorization layer
  inside pi.dev.
- **Confirmation (soft layer):** `~agent/.pi/agent/SYSTEM.md` instructs the
  model to ask the operator before any destructive or shared-state-affecting
  action. Not a sandbox; a footgun reducer.
- **API key:** read from `/etc/opsbridge/agent/api.key` (mode 0400, owned by
  `agent`). The launcher script reads it at session start and exports it as
  `ANTHROPIC_API_KEY` or `OPENAI_API_KEY`. Never hardcoded in the script.
- **Non-TTY guard:** the launcher exits 2 with a clear message if stdin is
  not a TTY, preventing non-interactive invocations (scripts, CI, port scans)
  from spawning pi.dev accidentally.

## 6. Configuration

Two files under `/etc/opsbridge/agent/`:

```toml
# /etc/opsbridge/agent/config.toml  (mode 0440, root:agent)
provider = "anthropic"        # "anthropic" | "openai"
model    = "claude-opus-4-7"  # bare model id (no provider prefix)
```

```
/etc/opsbridge/agent/api.key   (mode 0400, agent:agent)   — single line, the API token
```

The launcher script is regenerated by `opsbridge install` and `opsbridge config`
using these values. Model format passed to pi: `{provider}/{model}` (e.g.
`anthropic/claude-opus-4-7`).

### System prompt

```
~agent/.pi/agent/SYSTEM.md   (mode 0640, agent:agent)
```

Written at install time with OpsBridge's safety rules. Pi.dev reads this
path automatically on every session. Operators may edit it directly; running
`opsbridge install` will not overwrite an existing file.

Default content:

```markdown
You are an operations assistant running on a server. The user reached you via SSH.

Rules:
- Before running destructive commands (rm, drop, truncate, kill, overwrite), confirm with the user by asking explicitly.
- This host may have NOPASSWD sudo. Use it carefully.
- Be concise. Prefer direct answers and shell commands over lengthy explanations.
- When a task is ambiguous, ask one clarifying question rather than guessing.
```

### Credentials

File-based credentials (`~/.aws/credentials`, `~/.kube/config`, etc.) and
env-var credentials in `~/.profile` (sourced by every `bash` invocation) live
under `/home/agent/`. All are shared across every authorized operator key.

## 7. Admin CLI (`opsbridge`)

Console script installed in `/opt/opsbridge/agent/.venv`, symlinked to
`/usr/local/bin/opsbridge`. All subcommands require root.

| Subcommand | Behavior |
|------------|----------|
| `opsbridge install` | **Idempotent.** Creates the `agent` user, builds the opsbridge venv, installs pi.dev via npm, writes config files, generates the launcher script, writes the system prompt, wires sudoers + sshd. Pass `--reconfigure` to re-prompt for provider/model/key. `--skip-model-config` for unattended installs. |
| `opsbridge config` | Re-prompt for provider/model/key; regenerate the launcher script. |
| `opsbridge doctor` | Check agent user, opsbridge venv, pi binary, launcher script, config files, sudoers, sshd snippet, authorized_keys. Exit 0 ok, 1 error, 2 warning only. |
| `opsbridge enable` | Restore the sshd `ForceCommand` snippet and reload sshd. |
| `opsbridge disable` | Move the sshd snippet aside and reload sshd. |
| `opsbridge uninstall` | Remove the agent user, venv, config, launcher script, sudoers, and sshd snippet. Asks for explicit confirmation. |

## 8. Lifecycle journeys

### 8.1 Admin: first-time setup

```bash
curl -fsSL <release-url> | tar -xz -C /opt/opsbridge-src
sudo /opt/opsbridge-src/bootstrap.sh   # installs uv, builds venv, symlinks opsbridge
sudo opsbridge install --interactive
```

Interactive prompts:

```
[1/6] System user 'agent'... ok
[2/6] Python venv (opsbridge admin CLI)... ok
[3/6] Pi.dev agent... ok  (npm install @earendil-works/pi-coding-agent)
[4/6] Filesystem layout... ok
[5/6] Model config:
      Provider [anthropic/openai]: anthropic
      Model [claude-opus-4-7]:
      Paste API key (hidden): ********
      ok — launcher written to /usr/local/bin/opsbridge-agent
[6/6] sshd + sudoers... ok

Done. Add operator pubkeys to /home/agent/.ssh/authorized_keys.
```

Then authorize operators:

```bash
echo "ssh-ed25519 AAAA... alice@laptop" | sudo tee -a /home/agent/.ssh/authorized_keys
```

Verify:

```bash
sudo opsbridge doctor
```

### 8.2 Operator: first use

```
$ ssh agent@host
[pi.dev TUI opens]
> Show me the top 5 disk consumers under /var
[pi runs du, presents results]
> Is the nginx service running?
[pi runs systemctl status nginx]
> ^D (or pi's quit gesture)
```

### 8.3 Admin: routine operations

| Need | Command |
|---|---|
| Health check | `opsbridge doctor` |
| Maintenance window | `opsbridge disable` … `opsbridge enable` |
| Rotate API key or switch model | `opsbridge config` |
| Refresh pi.dev | `npm install -g --ignore-scripts @earendil-works/pi-coding-agent` |
| Add operator | append pubkey to `/home/agent/.ssh/authorized_keys` |
| Revoke operator | remove their pubkey line |

### 8.4 Admin: decommission

```bash
sudo opsbridge uninstall
```

## 9. Non-goals (v1)

- A web UI, REST API, or any non-SSH transport.
- Per-operator session isolation or credential namespacing.
- Tool sandboxing beyond Unix file permissions.
- Custom tool extensions (search, visit, ask, remember) — pi.dev's built-in
  tools cover the core use cases.
- Persistent cross-session preferences (pi.dev has its own session context).
- **Per-user CLI install mode** — the multi-operator service-account model
  behind a dedicated SSH identity is the product differentiator. If you want
  a smarter local terminal, use `aichat`, `ai-shell`, etc.

Runtime layout on a host (post-install):

```
/opt/opsbridge/agent/
└── .venv/                    # opsbridge admin CLI venv

/etc/opsbridge/agent/
├── config.toml               # provider, model
└── api.key                   # LLM API token

/usr/local/bin/opsbridge-agent   # generated launcher script (bash, mode 0550 root:agent)
/usr/local/bin/opsbridge         # symlink → /opt/opsbridge/agent/.venv/bin/opsbridge

/home/agent/
├── .ssh/authorized_keys
├── .profile                  # shell rc with opsbridge-agent launcher block
├── .bashrc                   # shell rc with opsbridge-agent launcher block
└── .pi/agent/
    └── SYSTEM.md             # OpsBridge safety rules (pi.dev reads this automatically)

/etc/sudoers.d/opsbridge-agent                       # NOPASSWD: ALL
/etc/ssh/sshd_config.d/50-opsbridge-agent.conf       # ForceCommand /usr/local/bin/opsbridge-agent
```

`pyproject.toml` declares one console script: `opsbridge = "opsbridge.admin:main"`.
Pi.dev is a separate npm package (`@earendil-works/pi-coding-agent`), installed
globally via `npm install -g`.

## 11. Open questions

- **Pi.dev version pinning.** Currently `npm install -g` fetches the latest.
  A future `opsbridge install` could pin a specific version and add it to
  `opsbridge doctor` checks.
- **Custom base URL / proxy.** Pi.dev supports custom model endpoints via
  `~agent/.pi/agent/models.json`. This is not managed by OpsBridge today;
  operators configure it directly.
- **Per-operator credential namespacing (v2).** Currently all operators
  authorized for `agent` share the same `/home/agent/` credential set.
